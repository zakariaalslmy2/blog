خطأ في Git: لم تكمل الدمج (MERGE_HEAD موجود)
واجهت رسالة الخطأ "fatal: You have not concluded your merge (MERGE_HEAD exists)" لأنك حاولت بدء عملية دمج جديدة بينما لا تزال هناك عملية دمج سابقة لم تكتمل. يحدث هذا عادةً عندما يؤدي الدمج إلى تعارض في الملفات، ويترك Git المستودع في حالة دمج معلق حتى يتم حل التعارض.
عندما تبدأ عملية دمج في Git، فإنه يقوم بإنشاء ملف خاص يسمى MERGE_HEAD لتتبع فرع الدمج.[1][2] إذا لم تكتمل عملية الدمج بنجاح، يبقى هذا الملف، مما يمنع عمليات الدمج الجديدة.[2]
لحل هذه المشكلة، لديك خياران رئيسيان: إكمال الدمج الحالي أو إلغاؤه.
الخيار الأول: إكمال عملية الدمج
إذا كنت ترغب في متابعة الدمج، اتبع الخطوات التالية:
تحقق من حالة الدمج: استخدم الأمر git status لعرض حالة المستودع الحالية. سيعرض لك هذا الملفات التي بها تعارضات.
code
Shell
git status
حل التعارضات: افتح الملفات التي بها تعارضات في محرر النصوص. سترى علامات مثل <<<<<<< و ======= و >>>>>>> التي تحدد الأجزاء المتعارضة من التعليمات البرمجية.[2][3] قم بتحرير هذه الملفات يدويًا لإزالة العلامات وحل الاختلافات بالطريقة التي تريدها.
إضافة الملفات التي تم حلها: بعد حل التعارضات في ملف، استخدم الأمر git add لإضافته إلى منطقة التجهيز، مما يشير إلى أن التعارض قد تم حله.
code
Shell
git add <اسم_الملف>
تنفيذ الدمج: بمجرد حل جميع التعارضات وإضافة جميع الملفات التي تم حلها، أكمل عملية الدمج عن طريق تنفيذ التغييرات.
code
Shell
git commit
سيفتح هذا محرر النصوص الافتراضي لكتابة رسالة تنفيذ للدمج. بعد الحفظ والإغلاق، سيتم إكمال الدمج.
الخيار الثاني: إلغاء عملية الدمج
إذا كنت لا ترغب في المتابعة في الدمج الحالي وتريد العودة إلى الحالة التي كان عليها المستودع قبل بدء الدمج، يمكنك إلغاء عملية الدمج.[1][4]
code
Shell
git merge --abort
هذا الأمر سيعيد مستودعك إلى الحالة التي كان عليها قبل محاولة الدمج، وسيزيل ملف MERGE_HEAD. بعد إلغاء الدمج، يمكنك محاولة دمج فرع api مرة أخرى إذا كنت ترغب في ذلك.
باختصار:
إذا كنت تريد إنهاء الدمج الذي بدأته، فقم بحل أي تعارضات، ثم استخدم git add و git commit.
إذا كنت تريد إلغاء الدمج والبدء من جديد، فاستخدم git merge --abort.

بما أنك تقوم فقط بإكمال عملية دمج (merge commit)، فأنت لست بحاجة إلى استعادة أي نص من المحاولة السابقة. الحل الأبسط هو حذف ملف الـ swap المؤقت والمحاولة مرة أخرى.

code

الخطوة 3: المحاولة مرة أخرى
الآن بعد أن تم حذف الملف المؤقت، قم بتنفيذ أمر الـ commit مرة أخرى. هذه المرة سيفتح المحرر بشكل نظيف بدون أي تحذيرات.
code
Shell

Git لا "يعدل" بل "يستبدل"
عندما تستخدم أمرًا لتعديل commit قديم (مثل git commit --amend أو git rebase)، فإن Git في الحقيقة لا يرجع للماضي ويمسح الـ commit القديم. بدلاً من ذلك، هو يقوم بالآتي:
يأخذ التغييرات من الـ commit القديم.
ينشئ commit جديد تمامًا بالرسالة الجديدة التي كتبتها.
يجعل الفرع الحالي (master في حالتك) يشير إلى هذا الـ commit الجديد.
يترك الـ commit القديم الأصلي مهجورًا في الخلفية.
تحليل ما حدث في مشروعك
بناءً على سجل git log الذي عرضته، هذا هو ما حدث على الأغلب:

دعنا نفصل الإجابة:
لماذا لا يعتبر عيبًا خطيرًا الآن؟
فرعك الرئيسي (master) نظيف ويحتوي على التاريخ الصحيح والمُحسّن.
يمكنك الاستمرار في العمل على فرع master وإضافة commits جديدة عليه دون أي مشاكل.
المشروع نفسه يعمل والكود سليم.
لماذا يعتبر ممارسة غير جيدة (عيبًا مستقبليًا)؟
هنا تكمن المشكلة الحقيقية:
مشاكل عند الدمج المستقبلي: هذه هي أكبر مشكلة. تخيل أنك بعد شهر من الآن، قمت بتطوير ميزة مهمة في فرع catgory وتريد دمجها في master. عندما تحاول الدمج، سيحاول Git دمج فرع يعتمد على تاريخ قديم (catgory) مع فرع يعتمد على تاريخ جديد (master). هذا قد يؤدي إلى:
تعارضات (Conflicts) غير متوقعة وصعبة الحل.
قد يحاول Git إعادة إدخال الـ commit القديم مرة أخرى إلى master، مما يسبب فوضى في تاريخ المشروع.
سيصبح شكل تاريخ المشروع معقدًا وغير مفهوم (يُعرف بـ "spaghetti history").
إرباك لك وللآخرين: أي شخص (بما في ذلك أنت في المستقبل) ينظر إلى سجل المشروع الكامل سيرى نسختين من نفس التغيير برسالتين مختلفتين. هذا يسبب إرباكًا ويجعل من الصعب تتبع التغييرات وفهم تاريخ المشروع بشكل صحيح.
تاريخ غير نظيف: الهدف من Git هو الحفاظ على سجل تاريخي نظيف وواضح ودقيق. وجود "commits مهجورة" مثل هذه يعتبر ضد هذه الممارسة.

لاحظ أن التاريخ والوقت متطابقان تمامًا في الاثنين!
ماذا يعني هذا؟
هذا يعني أن الطريقة التي استخدمتها لتعديل الرسالة كانت طريقة "ذكية" جعلت Git يحافظ على التاريخ الأصلي للعمل. هذا غالبًا ما يحدث عند استخدام أوامر متقدمة مثل git rebase -i واختيار reword (إعادة صياغة)، حيث يفهم Git أنك تغير "الوصف" فقط وليس "العمل" نفسه، فيحتفظ بتاريخ المؤلف الأصلي.
إذًا، للإجابة على سؤالك بوضوح:
عند التعديل هل يبقئ تاريخ الكوميت القديم او يستبدله بجديد؟
في حالتك أنت، بقي تاريخ الكوميت القديم ولم يتم استبداله. وهذا أفضل سيناريو ممكن، لأنه يحافظ على مصداقية تاريخ مشروعك. الكوميت الجديد والمُحسّن لا يزال يُظهر أنه تم إنشاؤه في 9 مارس 2025.
الخلاصة النهائية لمشكلتك:
هل فقدت تاريخك القديم؟ لا، الكوميت المُعدل يحمل نفس التاريخ الصحيح.
إذًا ما هي المشكلة المتبقية؟ المشكلة هي فقط وجود نسخة مكررة من هذا الكوميت (النسخة ذات الرسالة القديمة) والتي لا يزال فرع catgory يتذكرها.
ماذا سيفعل git rebase؟ كل ما سيفعله هو جعل فرع catgory ينسى النسخة المكررة القديمة، ويتذكر فقط النسخة الجيدة (التي تحمل تاريخك الصحيح والقديم).
يمكنك المتابعة بثقة تامة. أنت لن تخسر أي شيء من تاريخك، بل ستنظفه فقط.

بالتأكيد. سأرشدك خطوة بخطوة لتنفيذ هذا الأمر بأمان. العملية بسيطة جدًا.
تذكر أن ما سنفعله هو كأننا نقول لفرع catgory: "انتقل من الطريق القديم الذي يحتوي على الكوميت المكرر، وقفز إلى نهاية الطريق الجديد والمُحسّن الذي يسير عليه فرع master".
إليك الخطوات بالتفصيل. قم بتنفيذها أمرًا تلو الآخر في الطرفية (Command Prompt):
الخطوة 1: انتقل إلى الفرع الذي تريد إصلاحه
أولاً، يجب أن ننتقل إلى فرع catgory لأن هو الفرع الذي يعتمد على التاريخ القديم.
code
Shell
git checkout catgory
ماذا سيحدث؟ ستخبرك الطرفية أنها انتقلت بنجاح إلى الفرع:
Switched to branch 'catgory'
الخطوة 2: قم بتنفيذ عملية الـ Rebase
هذا هو الأمر الرئيسي. معناه الحرفي هو: "أعِد بناء هذا الفرع (catgory) ليكون أساسه (base) هو فرع master".
code
Shell
git rebase master
ماذا سيحدث؟
سيقوم Git تلقائيًا بأخذ كل الـ commits الفريدة في فرع catgory (التي لا توجد في master) ويضعها فوق آخر تحديث في master. بما أن catgory يعتمد على تاريخ قديم، فإن هذه العملية ستجعله يتجاهل هذا التاريخ القديم ويعتمد على التاريخ الجديد النظيف الموجود في master.
الناتج المتوقع سيكون شيئًا مثل:
Successfully rebased and updated refs/heads/catgory.
الخطوة 3: تحقق من النتيجة
الآن، دعنا نتأكد من أن كل شيء أصبح نظيفًا. اعرض سجل التاريخ مرة أخرى:
code
Shell
git log
ماذا سترى الآن؟
يجب أن تلاحظ أن الـ commit القديم ذو الرسالة السيئة قد اختفى تمامًا من السجل. لم يعد له وجود لأن لا يوجد أي فرع يشير إليه الآن. ستجد فقط النسخة الجيدة من الـ commit بتاريخه الصحيح (9 مارس).
الخطوة 4: ارجع إلى فرعك الرئيسي (اختياري ولكن موصى به)
لقد انتهينا من إصلاح catgory. من الجيد دائمًا العودة إلى فرع master لمواصلة عملك الرئيسي.
code
Shell
git checkout master
ماذا سيحدث؟ ستعود إلى فرع master:
Switched to branch 'master'
ملخص الأوامر للتنفيذ:
code
Shell
git checkout catgory
git rebase master
git checkout master
ماذا لو حدث تعارض (Conflict)؟
في حالتك، هذا غير محتمل، ولكن إذا حدث ورأيت رسالة "Conflict"، فلا تقلق. هذا يعني فقط أن Git لم يستطع تطبيق تغييراتك تلقائيًا. إذا لم تكن متأكدًا مما يجب فعله، يمكنك دائمًا إلغاء كل شيء والعودة إلى ما كنت عليه بأمان باستخدام الأمر:
code
Shell
git rebase --abort
هذا الأمر هو "زر الأمان" الخاص بك. سيلغي عملية الـ rebase ويعيد كل شيء كما كان قبل أن تبدأ.
